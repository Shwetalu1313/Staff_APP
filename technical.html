Technical Reference

Detailed implementation notes for the current staff portal, plus the backend work needed if this site becomes dynamic.

Current scope

Static multi-page frontend, browser-side state, client-side weather, and demo-only anonymous room behavior.

System

Current architecture overview

Current model: static HTML pages + one shared stylesheet + one shared script + browser-local state.

Static strengths

  • Simple hosting and deploy flow.
  • No required server runtime for normal browsing.
  • Fast initial load when assets stay lightweight.
  • Good fit for public navigation, documentation, link hub behavior, and browser-side layout personalization.

Static limits

  • No real shared user data without backend services.
  • No authenticated workflow state across devices.
  • Anonymous room is browser-local, not truly public.
  • Settings and content changes are client-side only.

Features

Interactive feature inventory

Theme system

Supports light, dark, and system mode. Theme preference is stored in browser storage, and resolved mode updates the site root data attribute.

Weather widget

Uses client geolocation when allowed, falls back to Yangon when blocked, and maps live weather code plus temperature into icon and color states.

Desk pet

Lightweight floating pet with local preference toggle, JSON-based reminder scripts, contextual messages, and reduced-motion safeguards.

Atom Voice

Anonymous room UI with local demo persistence, sender/right and other/left alignment, and no real cross-user synchronization yet.

AI chat demo

Local interactive chat with fun body-level commands like rotate, drop, and space for temporary visual effects.

Settings toggles

Clickable switches for user-facing preferences. Current behavior is browser-local and intended as a frontend foundation.

Section and nav controls

Each page section has per-section controls for hiding and drag sorting, and the top navigation links can also be drag-reordered. Tom opens the restore/reset panel for layout and nav customization.

Data

Storage, local state, and data sources

staff_theme Stores selected theme mode: light, dark, or system.
staff_settings Stores settings toggles like browser alerts, email digest, product updates, and desk pet visibility.
staff_layout_[page] Stores per-page hidden sections and drag-sorted order so each page layout can be personalized in the current browser.
staff_nav_order Stores the user-defined top navigation order for the current browser.
public_room_messages Stores Atom Voice demo messages in the local browser profile only.
assets/pet-scripts.json Holds structured reminder scripts by condition so future content can be expanded without touching logic.
Open-Meteo Client-side weather source for current conditions and icon/color mapping.
Important limitation:

All current persistence is local to the browser. Nothing here is authoritative user data until a backend system is introduced.

Developer UI

Reusable classes for buttons, forms, cards, and layout

Use existing classes first. The goal is consistency, not one more custom pattern.

1. Pick the surface Start with panel for the container.
2. Pick the layout Use container, section, or grid helpers like cards-2.
3. Pick the action style Use btn, icon-btn, or chips based on priority.
4. Keep labels visible Buttons, forms, and switches still need clear text or accessible names.

Page and layout classes

  • section and section-standard create the normal vertical rhythm used across pages.
  • container keeps content width aligned with the header and footer.
  • panel is the primary surface wrapper for cards, content blocks, chat panels, settings groups, and docs modules.
  • cards-2, cards-3, and related grid helpers should be reused before introducing a custom grid.

Action and status classes

  • btn is the main action pattern. Use modifiers that already exist before creating a new button visual.
  • icon-btn is for compact circular icon actions in the header and utility areas.
  • chip, profile-chip, and similar compact pills work for small contextual metadata rather than primary actions.
  • icon-badge and count badges should only appear when there is meaningful state. Avoid showing 0 when nothing needs attention.

Form classes

  • settings-option, switch wrappers, and labeled control rows should be reused for preference forms.
  • Inputs should stay paired with visible labels or accessible names even when the UI is visually compact.
  • Keep submit actions attached to the field group they control. Do not place unrelated actions inside the same form row.
  • For future dynamic forms, introduce validation message classes once backend validation rules are defined.

Content block classes

  • doc-card, doc-table, and doc-row already support documentation-style content.
  • Chat surfaces should preserve message-group structure rather than mixing generic cards and chat bubbles.
  • Feature-specific surfaces such as weather, AI, and pets should remain isolated in their own wrappers so styles do not leak into global card rules.
Use this for a button btn for primary action
Use this for an icon action icon-btn in tight spaces
Use this for a card panel as the main surface
Use this for a form row settings-option or a labeled row 
<section class="section section-standard">
  <div class="container cards-2">

    </article>

</section>
Developer rule:

If a new page needs a button, form row, panel, or grid, first search styles.css for an existing class. Matching the current visual language is better than adding one more near-duplicate selector.

JavaScript

Shared script responsibilities and function-level guidance

main.js is the shared runtime. Add new code by feature, not as random global helpers.

Boot Initialize global UI and page hooks
Feature setup Theme, weather, pets, docs, chat
State Read and write browser storage
UI update Reflect state back into the DOM
Boot and page setup Initializes common UI behavior, page-specific features, keyboard shortcuts, and theme state after the DOM is ready.
Theme functions Resolve light, dark, or system, update root data attributes, and persist the user preference in staff_theme.
Weather functions Request geolocation when allowed, fall back to Yangon, call Open-Meteo, then map weather code plus temperature into icon and color states for the nav widget.
Pet functions Load JSON reminder scripts, evaluate conditions such as long sitting or cold weather, display bubble text, and keep animations lightweight.
Layout manager functions Attach hide and drag controls to page sections, manage restore/reset behavior, and persist per-page layouts in staff_layout_[page].
Navigation customization Lets users drag the shared top-nav links into a custom order and stores that preference in staff_nav_order.
Docs page functions Handle left-topic activation and client-side search filtering for documentation sections.
Chat/demo functions Support the home and dedicated AI chat UI plus the Atom Voice local room demo and visual easter-egg commands.

How to add a new function

  • Keep it scoped to one feature domain if possible.
  • Use explicit names that describe the side effect, not vague helpers.
  • Prefer reading page context from data-page or feature selectors rather than guessing from the URL text.
  • Persist only the minimum browser state needed for the feature.

How to refactor later

  • Split global infrastructure from feature modules first.
  • Move weather, pets, docs, AI, and layout management into separate files or modules.
  • Create one stable storage utility so keys and reset behavior stay centralized.
function setupFeature() {
  const page = document.body.dataset.page;
  if (page !== "settings") return;

  const toggle = document.querySelector("[data-setting-toggle]");
  if (!toggle) return;

  toggle.addEventListener("click", handleToggle);
}

CSS

How the stylesheet is organized and how to extend it safely

styles.css is both the design system and page stylesheet. Add styles near the related feature block, not at random.

Tokens Theme colors and shared variables
Base UI Typography, body, shared elements
Components Header, footer, cards, buttons
Feature blocks AI, pets, weather, docs, settings

Core CSS layers

  • Theme variables and root-level color tokens.
  • Base typography, body background, and generic element rules.
  • Shared components such as header, footer, cards, chips, and buttons.
  • Feature/page modules such as AI, weather, pets, Atom Voice, settings, and documentation.
  • Responsive breakpoints for tablet and phone adaptation.

Safe extension rules

  • Add new selectors near the closest existing feature block instead of scattering them randomly.
  • Prefer class selectors over element-heavy chains to keep overrides predictable.
  • Use CSS variables for color and surface changes when the value may need dark-mode support later.
  • Avoid universal transitions and expensive blur-heavy effects during scroll.

Responsive expectations

  • New components should be checked at desktop, tablet, and narrow mobile widths.
  • Fixed widgets such as pets, weather popovers, and action bubbles must not overlap critical content on small screens.
  • If a layout needs equal-height panels, keep the content scrollable inside the panel rather than forcing the entire row taller.

Animation rules

  • Prefer transform and opacity animations.
  • Keep decorative motion subtle and never tie broad UI invalidation to scroll state.
  • Respect prefers-reduced-motion for pets and non-essential animated effects.
.new-widget {
  border-radius: 18px;
  background: var(--panel-bg);
  color: var(--ink-800);
}

  border-color: rgba(126, 178, 255, 0.16);
}

Dynamic Future

What must connect to backend services

If this portal moves from static to dynamic, the frontend already suggests several product areas that should no longer depend on browser-local placeholders. The items below are the minimum backend domains to plan.

Authentication and identity

  • SSO or company identity integration.
  • User session handling and token refresh.
  • Role-based access control for staff, admin, and reviewer views.
  • Profile data source for avatar, name, department, and permissions.

Requests and queue data

  • Services catalog should be driven by backend-configured teams and forms.
  • Workspace queue counts should come from real approvals, tickets, and tasks.
  • Support desk readiness should come from operational metrics or ticketing APIs.

Atom Voice real-time room

  • Database-backed message storage.
  • Real-time transport via WebSocket or server-sent events.
  • Moderation rules, rate limiting, abuse controls, and retention policy.
  • Optional anonymous session identifiers without exposing personal identity.

Settings sync

  • User preferences API for theme, notifications, and pet visibility.
  • Cross-device sync so settings remain stable outside one browser.
  • Notification preference mapping to email, browser push, and in-app delivery.

Notification service

  • Unread notifications endpoint.
  • Read/unread state and notification categories.
  • Browser push infrastructure if real opt-in is needed.

AI assistant integration

  • Secure chat endpoint for prompts and responses.
  • Guardrails for allowed topics and access boundaries.
  • Conversation logging policy and retention design.

Content management

  • FAQ, version notes, legal, privacy, and documentation content should ideally move into an admin-managed content source.
  • Release notes and footer links should be editable without manual HTML changes.
  • User-personalized section layouts should move to server-backed user preferences if cross-device consistency is required.

Operational controls

  • Audit logs for profile actions, preference changes, and moderation events.
  • Error monitoring and analytics.
  • Feature flags to release pet, AI, chat, or docs features safely.
Recommended order:

Identity, services/queue APIs, and notifications should come first. Real-time Atom Voice, admin-managed content, and AI chat can follow after access control and data ownership are stable.

Roadmap

Phase-by-phase plan to move from static to dynamic

Do not rebuild everything at once. Replace local-only behavior in phases.

Phase 1

Main feature delivery

Make the portal truly usable with real identity, queue data, services, notifications, settings, and content APIs.

Phase 2

Collaboration and content

Add real Atom Voice, content management, support metrics, and role-based visibility.

Phase 3

Intelligence and optimization

Add secured AI, analytics, moderation tooling, feature flags, and richer personalization.

Phase 1 detailed must-haves

  • Company login and session lifecycle.
  • User profile API with role and permission data.
  • Real queue/task counts for the workspace.
  • Service catalog and request form submission APIs.
  • Notification retrieval plus read state.
  • Persistent settings sync across devices.
  • Basic content service for homepage summaries and important portal copy.

Why Phase 1 comes first

  • These features affect daily utility, trust, and repeat use.
  • Without identity and real data, the portal stays mostly decorative.
  • Later AI or chat improvements are weaker if the core operational data is still fake or local-only.

Suggested backend deliverables

  • API gateway or app server with authenticated endpoints.
  • Relational database for users, requests, notifications, preferences, and content.
  • Role model and audit logging from the beginning.
  • Staging environment with seeded test data and monitoring.

Frontend migration approach

  • Replace localStorage reads feature by feature, not all at once.
  • Keep the current UI contracts stable while wiring real APIs behind them.
  • Document each API payload before implementation so future pages can reuse the same data model.
How to execute the roadmap:

Start with auth and real data contracts, keep the current UI stable, then replace browser-local placeholders feature by feature instead of shipping a disruptive redesign.

Design

Visual system and layout rules

Typography

Manrope handles dense UI text while Space Grotesk gives section titles a more branded voice. This split keeps the interface readable without looking generic.

Surface style

Most panels use soft gradients, rounded corners, low-contrast borders, and restrained shadows rather than heavy glassmorphism. This helps maintain polish without causing heavy scroll repaint cost.

Light mode direction

Light mode intentionally carries more color and visible texture so the site avoids flat corporate white. Decorative motion was reduced earlier to improve scroll smoothness.

Responsive approach

Layouts progressively collapse from multi-column boards to stacked panels, with dedicated rules for header wrapping, chat widgets, app grids, and fixed utility elements.

Palette

Color palette and UI roles

The interface uses a soft multi-accent palette rather than a single-brand monotone. The goal is to keep the product warm and usable in light mode while still supporting dark mode cleanly.

Accent 1

Primary actions, links, active pills, weather cool states, and structured UI emphasis.

Accent 2

Success states, wellness notes, positive cards, and selected support-friendly accents.

Accent 3

Warm highlights, attention markers, weather warmth, and texture variation in light mode.

Accent 4

Urgency, caution, legal/privacy emphasis, and occasional contrast support for key badges.

Background texture Static layered gradients and subtle pattern overlays are preferred over constant background animation.
Dark mode Maintains the same accent logic with darker surfaces and lifted text contrast instead of oversaturated neon treatment.
Accessibility Accent use should continue to pair with text labels and not rely on color alone for meaning.

Quality

Performance, accessibility, and maintainability notes

Performance

  • Continuous decorative motion was reduced to improve scroll behavior.
  • Transform/opacity are preferred over layout-changing animation.
  • Heavy blur and broad universal transitions should stay limited.

Accessibility

  • Keyboard shortcuts avoid firing while typing in inputs.
  • Form controls include labels or accessible names.
  • Reduced-motion handling is already relevant for the desk pet and animated UI areas.

Maintainability

  • Shared HTML fragments are currently duplicated page by page.
  • A component or template system would reduce repeated nav/footer edits.
  • Version notes should stay aligned with footer counts and page inventory.

Testing gaps

  • No automated browser regression suite currently exists.
  • Interactive behavior depends on manual verification in viewport/device testing.
  • Search and docs topic filtering should be checked on mobile after each structural docs change.
